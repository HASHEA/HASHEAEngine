# AshEngine 底层引擎改进与需求文档

> 本文档只基于当前实际参与编译的主线：
> `project/src/engine`
>
> 目标不是重写一套“更大”的底层，而是把当前运行时骨架打磨成一个能长期支撑编辑器、内容生产和后续功能扩展的稳定平台。

## 1. 当前底层状态结论

当前底层已经具备以下雏形：

- 基础设施层：日志、内存、文件、窗口、服务
- 图形抽象层：`GraphicsContext`、`Swapchain`、`Buffer`、`Texture`、`Shader`
- 功能层：`Application`、`Renderer`、`RenderDevice`
- Vulkan 后端实现

但从“长期支撑 Unity 风格编辑器和复杂项目”的标准来看，当前底层仍处于：

**能启动、能渲染、能继续演进，但还没有形成稳定可持续的 runtime 平台。**

## 2. 当前主要问题判断

以下是我认为最需要优先改进的点。

### 2.1 工程健康度不够稳定

当前代码里已经能看到一些明显的未完成或不一致实现，例如：

- `ShaderCache` 文件读写接口几乎是空实现
- `Serializer` 中存在明显的命名不一致、大小写不一致和疑似未完成模板代码
- 若后续依赖这些模块，会直接影响底层稳定性

这说明当前工程更像“快速搭建中的骨架”，还没有到可以把所有上层都压上去的程度。

### 2.2 平台层与应用层耦合仍然偏重

例如窗口大小变化时，`WindowWin` 直接访问 `Application::get_swapchain()` 并触发 `resize_swapchain()`，说明窗口事件处理还没有经过统一事件总线或平台消息分发层。

这会带来几个问题：

- 平台层不独立
- 编辑器多窗口支持困难
- 输入系统和 UI 系统难以扩展
- 生命周期边界不清晰

### 2.3 Service 系统太轻，缺依赖管理能力

当前 `Service` 只有 `init/shutdown`，`ServiceManager` 只是一个简单哈希表注册器。

这对 demo 足够，但对大型引擎不够：

- 没有依赖顺序描述
- 没有生命周期阶段
- 没有线程安全约束
- 没有运行时状态监控
- 没有服务配置管理

### 2.4 序列化系统需要重做级别的整理

现有 `hserialization.h/.cpp` 暴露出来的问题不只是“功能少”，而是存在明显的实现未收口迹象。

如果未来要支持：

- Scene
- Prefab
- Asset Meta
- Project Settings
- Editor Layout
- Shader Cache Meta

那么底层必须有一个稳定、可测试、版本化、可迁移的序列化方案。

### 2.5 缺少正式资源系统

当前有 GPU 资源创建接口，但还没有形成真正的运行时资源层：

- 没有统一资源句柄
- 没有资源生命周期状态机
- 没有异步加载流程
- 没有依赖关系管理
- 没有资源缓存和失效机制
- 没有导入产物与运行时资源的边界

这会直接卡住编辑器资源浏览器、导入器、Prefab、热重载等系统。

### 2.6 缺少反射系统

未来编辑器的 Inspector、序列化、脚本桥接、组件编辑、命令系统，都需要反射元数据支撑。

如果底层没有统一反射层，上层会很快陷入手写 UI、手写序列化、手写注册表的重复劳动。

### 2.7 缺少场景运行时模型

当前引擎底层还没有正式的：

- Scene
- Entity
- Component
- World
- System

等核心运行时模型。

这意味着编辑器即使 UI 做出来，也缺一个可编辑、可保存、可运行的统一数据宿主。

### 2.8 渲染抽象还不够“平台化”

当前 `Renderer` 的雏形是好的，但还没有形成面向大型项目的渲染平台能力：

- 没有 Render Graph
- 没有统一 Frame Graph 资源管理
- 没有多视口/多相机调度框架
- 没有材质系统
- 没有渲染特性注册机制
- 没有正式的 debug label / GPU marker / pass profiler 抽象

这会限制后续编辑器视口、材质预览、场景预览、离线预览和后处理组合能力。

### 2.9 Shader 编译与缓存系统未闭环

当前底层已经有 Shader、DXC、ShaderCache 相关结构，但“编译、缓存、失效、热重载、依赖 include 追踪”这一整条链路还没有真正闭环。

这会直接影响：

- 编辑器热重载体验
- 材质开发体验
- 启动速度
- CI 构建稳定性

### 2.10 缺少任务系统和异步基础设施

未来引擎和编辑器都会需要：

- 资源导入
- 缩略图生成
- Shader 编译
- 文件扫描
- 场景加载
- 后台构建

如果底层没有统一 job/task/async pipeline，很快会到处出现临时线程和难维护的状态同步代码。

### 2.11 缺少引擎级输入与事件系统

当前窗口层有 GLFW callback，但并没有形成正式输入系统：

- 没有 Action / Axis 映射
- 没有事件分发
- 没有 UI 输入路由
- 没有编辑器快捷键上下文
- 没有输入录制/回放

这会影响运行时与编辑器两侧。

### 2.12 缺少测试、诊断与质量基线

未来如果没有这些底层保障，扩展速度越快，回归成本越高：

- 单元测试
- 序列化一致性测试
- Shader 编译测试
- 资源加载测试
- GPU 能力探测与日志
- 崩溃转储与错误上下文

## 3. 底层改进总目标

底层演进目标应当是：

1. 成为编辑器唯一可信赖的运行时平台
2. 保持清晰分层，避免平台层、渲染层、编辑器层互相反向依赖
3. 先解决“稳定性与扩展性”，再追求“功能数量”
4. 把未来高频会重复造轮子的能力提前沉到 runtime

## 4. 架构原则

### 4.1 平台层独立

- 窗口、输入、文件系统、时间、线程、原生平台能力应独立于应用逻辑
- 平台事件不应直接调用上层业务对象

### 4.2 运行时与编辑器边界清晰

- 引擎底层提供稳定 API
- 编辑器不直接操纵 Vulkan 细节
- 编辑器扩展不能污染平台层和 RHI 层

### 4.3 数据驱动优先

- Scene、Asset、Settings、Prefab、Material 都应建立可序列化的数据模型
- 运行逻辑和展示逻辑分离

### 4.4 任务与资源统一调度

- 后台任务、资源加载、热重载、导入构建都应走统一系统

### 4.5 先保证工程健康，再继续堆功能

- 能编译
- 可测试
- 可追踪
- 可恢复

这些优先级高于新增高级特性。

## 5. 底层需求列表

以下按阶段和优先级列出。

---

## 5.1 P0：工程健康与平台稳定阶段

目标：

先把底层做成可靠地基。

需求：

- 修复并清理当前未收口模块
- 建立最小编译健康基线
- 整理命名规范、错误处理规范、断言规范
- 建立统一日志分类
- 建立最小单元测试框架
- 为关键模块增加 smoke test
- 整理 `Application` 生命周期边界
- 解除平台层对上层业务对象的直接调用
- 让窗口 resize、输入等事件进入统一事件分发层
- 建立正式输入系统骨架
- 建立平台消息队列或事件总线

交付标准：

- 引擎底层可稳定编译
- 平台事件流清晰
- 输入、窗口、生命周期边界明确
- 关键基础模块具备最小自动验证

---

## 5.2 P1：数据与资源基础阶段

目标：

让底层具备支撑编辑器的基础数据能力。

需求：

- 重构序列化系统
- 支持版本化序列化
- 支持向前兼容与数据迁移
- 建立反射系统基础能力
- 支持类型注册、字段描述、枚举描述、容器描述
- 建立统一资源句柄系统
- 建立资源状态机：
  - Unloaded
  - Loading
  - Loaded
  - Failed
  - Unloading
- 建立异步资源加载框架
- 建立资源缓存和引用计数策略
- 建立资源依赖图
- 建立文件变更监听与资源失效机制
- 建立 Asset Runtime/Import Data 边界

交付标准：

- Scene / Prefab / Meta / Settings 等都能基于统一底层数据机制实现
- 资源加载和编辑器资源浏览不再依赖临时方案

---

## 5.3 P2：运行时世界模型阶段

目标：

形成一个真正可以承载游戏与编辑器场景的运行时模型。

需求：

- 建立 `World`
- 建立 `Scene`
- 建立 `Entity`
- 建立 `Component`
- 建立 `System`
- 形成统一 Tick / Update / Render 阶段
- 支持运行时世界与编辑器世界隔离
- 支持对象创建、销毁、复制、克隆
- 支持对象唯一标识
- 支持层级关系与场景节点
- 支持序列化与反序列化

交付标准：

- 编辑器和运行时都能基于同一套场景底座工作
- 场景对象不再散落在各处临时结构中

---

## 5.4 P3：渲染平台升级阶段

目标：

让渲染层从“能提交 DrawCall”升级为“可组织复杂图形工作流的平台”。

需求：

- 建立 Render Graph
- 建立 Frame 级资源生命周期管理
- 建立 Pass 注册与执行模型
- 建立多视口 / 多相机渲染支持
- 建立材质系统基础抽象
- 建立渲染特性模块化机制
- 建立 GPU marker / debug label / pass profiler 抽象
- 建立渲染统计接口
- 建立统一屏幕空间、离屏、预览渲染路径
- 建立 shader include 依赖追踪
- 建立 shader cache 真正落盘与失效机制
- 建立 shader 热重载链路

交付标准：

- 编辑器视口、材质预览、离屏预览和后处理可以复用同一渲染底层
- 渲染管线不再靠手写线性提交硬编码扩展

---

## 5.5 P4：异步、任务与大工程支持阶段

目标：

让引擎底层具备扩展到更大项目的能力。

需求：

- 建立 Job System
- 建立主线程/后台线程任务调度
- 建立依赖任务链
- 建立线程安全资源提交规范
- 建立后台导入和后台编译支持
- 建立文件扫描与目录监听服务
- 建立缓存构建管线
- 建立统一 profiling 点位
- 建立崩溃收集和运行时诊断基础能力

交付标准：

- 底层能承载资源导入、缩略图、shader 编译、场景加载等后台工作
- 不再依赖临时线程逻辑拼接功能

## 7. 推荐执行顺序

建议按下面顺序推进，而不是并行大面积开坑：

1. 工程健康清理
2. 平台事件与输入系统
3. 序列化与反射重构
4. 资源系统
5. Scene / World 运行时模型
6. Shader 编译缓存闭环
7. Render Graph
8. Job System 和后台任务

## 8. 当前建议的工程决策

从当前状态出发，我建议：

- 不要急着继续往底层塞更多功能点
- 先把“基础设施是否可靠”解决掉
- 底层优先服务编辑器骨架建设
- 每新增一个系统，都要求：
  - 有明确边界
  - 有测试入口
  - 有可序列化模型
  - 有生命周期定义

## 9. 一句话结论

当前底层最该补的不是某个炫酷特性，而是：

**平台边界、数据基础、资源体系、任务体系、渲染组织能力。**

这几项一旦补稳，后面的编辑器、材质系统、Prefab、Play Mode、插件系统都会顺很多。
