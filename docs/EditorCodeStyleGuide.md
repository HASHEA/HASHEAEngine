# AshEngine Editor Code Rules

> 适用范围：
> - `project/src/editor`
>
> 目标：
> - 这份文档给 AI / 代码代理直接使用。
> - 只保留“改代码时可直接执行”的规则。
> - 不讲背景，不讲历史，不重复解释。

---

## 1. 最高优先级

- 不要继续扩大 `EditorApplication` 的职责。
- 不要继续扩大 `EditorContext`。
- 不要把可撤销编辑操作写成 panel 直改 scene。
- 不要在头文件里放 `main()`、启动流程实现、工作目录初始化或大段非模板实现。
- 不要新增长期裸指针生命周期风险。
- 不要让头文件依赖 PCH 才能编译。
- 不要把同一个 helper 在多个 `.cpp` 再复制一遍。

---

## 2. 修改现有代码时的默认策略

- 先沿用现有结构；如果现有结构明显失控，优先做小步收口，不继续堆功能。
- 优先把逻辑放到正确层级：
  - `Panel`：显示、输入收集、调用服务
  - `Service`：领域逻辑
  - `Command`：可撤销修改
  - `Controller / Coordinator`：跨模块编排
  - `EditorApplication`：启动、装配、主循环编排
- 如果一个改动需要同时碰 UI、状态、持久化、事件、命令，先考虑拆 helper / controller。

---

## 3. 依赖与生命周期

- 必需依赖优先用引用或明确的非空约束，不要默认做成可空裸指针。
- 可选依赖才用裸指针或可选引用。
- owner 用 `std::unique_ptr`。
- observer 不要长期缓存；能临时查就临时查。
- 如果必须缓存 observer 指针，必须满足：
  - owner 明确
  - 生命周期稳定
  - 失效点单一
- 不要靠大量 `if (!service)` 掩盖设计问题。

---

## 4. Panel / Service / Command

### Panel

- 负责绘制 UI。
- 负责收集输入。
- 可以持有轻量 UI draft state。
- 不负责复杂领域编排。
- 不直接拥有领域对象所有权。
- 可撤销修改必须走 `Command` + `UndoRedoService`。

### Service

- 不带 UI 细节。
- 不把持久化、运行态句柄、领域规则全揉在一起。
- 对外暴露稳定语义，不暴露内部容器细节。

### Command

- 必须尽量可逆。
- `execute / undo` 失败语义必须稳定。
- selection 变化要显式表达。
- 不要偷偷依赖宽上下文中的无关服务。

---

## 5. 头文件规则

- 头文件只放声明；非模板实现放 `.cpp`。
- 头文件优先前置声明，减少 include。
- `.cpp` 第一行优先 include 自己的头文件。
- include 顺序：
  1. 当前文件对应头
  2. 同模块工程头
  3. 其他工程头
  4. 第三方头
  5. 标准库头
- 不同 include 组之间空一行。
- 头文件中禁止：
  - `using namespace`
  - 文件级 `using std::string;`
- 即使启用了 PCH，也要显式 include 自己直接使用的标准库头。

### 5.1 PCH 规则

- 当前 Editor 的 PCH 文件是：
  - `project/src/editor/EditorPCH.h`
  - `project/src/editor/EditorPCH.cpp`
- `PCH` 只用于编译加速，不用于隐藏真实依赖。
- 头文件必须在“不依赖 PCH”的前提下自洽。
- `.cpp` 即使有 `forceinclude`，也优先显式 include 自己直接使用且语义关键的头。

以下情况才考虑把头加入 `EditorPCH.h`：

- 被大量 `.cpp` 稳定使用
- 很少改动，接口相对稳定
- 解析成本高，但属于基础设施依赖
- 明显属于 Editor 全局通用依赖，而不是某个局部模块私货

适合进入 `PCH` 的典型内容：

- 稳定的标准库头
- 稳定的第三方基础头
- Editor 几乎所有模块都会用到的基础 Engine / Editor 头

以下情况不要为了省事把头加入 `EditorPCH.h`：

- 只被一两个 `.cpp` 使用
- 刚引入、还在频繁改
- 某个 panel / service / feature 私有的头
- 改动频繁、加入后会放大整体重编译范围的头
- 只是因为当前文件缺 include，想靠 `PCH` 临时补洞

以下情况通常应该补充 `PCH`：

- 同一个重型头已经在很多 `.cpp` 重复出现
- 它已经成为 Editor 多数模块的共同基础依赖
- 加入 `PCH` 后不会模糊模块边界，也不会把局部实现依赖变成全局默认依赖

判断原则：

- “很多文件都 include 了”还不够；
- 还要满足“稳定、通用、低风险、不会扩大耦合”。

修改 `EditorPCH.h` 后要额外检查：

- 是否让某些头文件继续错误地依赖了 `PCH`
- 是否把局部模块依赖错误提升成了全局依赖
- 是否引入了本不该全工程传播的重型头

### 5.2 编码规则

- 新文件默认使用 `UTF-8`。
- 源码、脚本、文档默认优先：
  - `UTF-8 without BOM`
- 不要在同一目录或同一模块里混用多种文本编码。
- 不要为了顺手保存，把无关文件的编码一起改掉。
- 修改已有文件时，默认保持它当前可正常工作的编码；只有在本次任务明确要统一编码时才改。

以下文件默认按 `UTF-8 without BOM` 处理：

- `.h`
- `.cpp`
- `.inl`
- `.md`
- `.lua`
- `.txt`

以下文件处理规则：

- `.json`
  - 默认也使用 `UTF-8`
  - 若现有文件已能被当前工具链稳定读取，非必要不因为“规范化”单独改编码
- `.ini`
  - 默认使用 `UTF-8`
  - 若现有解析路径已经兼容 `BOM`，也不要因为顺手修改把文件来回切编码

什么时候应该补一次编码整理：

- 新增文件出现乱码风险
- 同一模块内已经混入两种以上编码
- 构建脚本、配置解析、日志输出、UI 文案出现编码问题
- 文本文件被错误工具保存后，已经影响 diff、解析或运行

什么时候不要改编码：

- 只是顺手格式化
- 只是改一两行逻辑
- 文件当前编码虽然不理想，但没有在本任务里造成问题
- 改编码会让 diff 大量噪音化，掩盖真实逻辑改动

非 ASCII 文本规则：

- 标识符、宏、配置 key、序列化 key 默认继续使用 ASCII。
- 用户可见文案、文档内容、必要注释可以使用中文。
- 如果文件原本全是 ASCII，非必要不要因为一条临时注释引入中文。

## 6. 命名规则

- 类型、枚举、类型别名：`PascalCase`
- 函数、方法：`PascalCase`
- 局部变量、参数、普通成员变量：`lowerCamelCase`
- 私有成员变量：`_lowerCamelCase`
- 常量：`kName`
- 宏：`ALL_CAPS`

额外规则：

- 类型名用名词。
- 函数名用动词或动宾结构。
- 不要让名字看起来像纯 helper，实际却做副作用编排。
- bool 名字优先用：
  - `isXxx`
  - `hasXxx`
  - `canXxx`
  - `shouldXxx`
  - `allowXxx`
  - `enableXxx`

### 6.1 变量前缀规则

使用类型前缀减少阅读成本。分为**必须**和**推荐**两级。

#### 必须使用的前缀

- `bXxx`：`bool`
- `strXxx`：`std::string`
- `pXxx`：裸指针
- `upXxx`：`std::unique_ptr`
- `spXxx`：`std::shared_ptr`

#### 推荐使用的前缀

以下前缀在显式类型声明下冗余度较高，推荐但不强制：

- `iXxx`：有符号整数
- `uXxx`：无符号整数
- `fXxx`：浮点数
- `svXxx`：`std::string_view`
- `refXxx`：引用语义变量或明确借用对象
- `vecXxx`：`std::vector`
- `mapXxx`：map / unordered_map
- `setXxx`：set / unordered_set
- `arrXxx`：数组或 `std::array`
- `optXxx`：`std::optional`
- `eXxx`：枚举值

规则：

- 私有成员在类型前缀前再加 `_`，例如：
  - `_bInitialized`
  - `_pViewportPanel`
  - `_strActiveDirectoryPath`
- 不要同一文件里混用“有前缀”和“无前缀”两套变量风格。
- 前缀只解决类型识别，不代替语义命名；名字后半段仍要表达含义。

---

## 7. 枚举、bool、常量

- 默认优先使用 `enum class`。
- 枚举值使用 `PascalCase`。
- 0 值必须显式定义，常用：
  - `None`
  - `Unknown`
  - `Default`
- flags 继续沿用工程现有 `Flags / FlagBits` 风格。
- 多个 bool 一起表达一个状态时，优先改成枚举或 options struct。
- 避免含义不清的 bool 参数，比如 `foo(true, false, true)`。
- 重复出现或有语义的字面量提成 `constexpr`，不要继续裸写 magic number。
- 默认优先显式写出变量类型，不要依赖 `auto` 推导。
- 以下场景允许使用 `auto`：
  - 迭代器类型（如 `auto it = container.find(key)`，类型极长且无信息量）
  - `std::make_unique` / `std::make_shared` 返回值（右侧已明确类型）
  - structured binding（如 `auto& [key, value]`）
- 除以上场景外，不要为了省代码使用 `auto`。

---

## 7.5 const 规则

- 方法如果不修改对象状态，必须标 `const`。
- 局部变量如果赋值后不再修改，优先标 `const`。
- 指针参数如果不修改指向对象，优先用 `const T*`。
- 不要为了"以后可能修改"而提前去掉 `const`。

---

## 8. 性能规则

- 能用 `const T&` 的输入参数，不要按值传递大对象。
- 以下类型默认优先用 `const&`：
  - `std::string`
  - `std::vector`
  - `std::array`
  - map / set 容器
  - 自定义 struct / class
- 小型标量按值传递：
  - `bool`
  - 整数
  - 浮点数
  - 枚举
  - 轻量 handle
- 只在“明确需要接管所有权”时按值接收并 `std::move` 到成员。
- 只读返回值优先：
  - `const T&`
  - 指针
  - `std::string_view`
- 不要无意义复制容器、字符串和大 struct。
- range-for 默认写成：
  - `const T& item`
  - `T& item`
  不要默认按值遍历。
- 能 `reserve` 的容器提前 `reserve`。
- 循环里不要重复做：
  - 字符串拼接
  - 路径组装
  - 容器拷贝
  - 多次相同查询
- 热路径不要频繁构造临时 `std::string`。
- 能复用对象时优先复用，不要每帧反复 new 临时对象。
- 不要为了省代码在非允许场景使用 `auto`；显式类型优先，避免隐藏拷贝和错误推导。
- 返回大对象前，先确认是否真的需要返回拥有值，而不是返回引用、视图或写入输出参数。

---

## 9. 日志与注释

### 日志

- 以下情况应该写日志：
  - 用户可感知失败
  - 文件读写失败
  - 场景加载、保存、重载失败
  - 命令执行 / undo / redo 失败
  - 面板、服务、会话初始化或 shutdown 失败
  - 进入 fallback、降级路径或兼容分支
  - 一次性的关键状态切换：
    - scene changed
    - layout reset
    - viewport create / destroy / rebind
    - 关键资源缺失后禁用某功能
- 以下情况通常也应该写一条 `Info` 或 `Warning`：
  - 自动修正了不合法输入或旧数据
  - 忽略了一次用户请求，且用户可能误以为已成功
  - 因配置、缺资源、缺上下文而跳过某能力
- 用户可感知失败：只在合适边界记录一次，且带上下文。
- 热路径、每帧路径默认不要打 `Info` 流水日志。
- 成功日志只保留“低频且关键”的一次性事件，不要把正常路径打成流水账。
- 临时 trace 要有明确清理意图，问题定位后删掉。
- 不要沿调用栈层层重复打印同一个失败。
- 如果底层已经记录了失败，上层默认只补充“这次操作因此失败了什么”，不要再重复错误细节。
- 日志尽量带这些上下文中的相关项：
  - action id
  - entity id
  - scene path
  - viewport id
  - 数量 / 尺寸
- 日志文案要回答这几个问题中的至少两个：
  - 什么操作失败/跳过了
  - 影响了什么
  - 原因是什么
  - 关联对象是谁
- 选择级别的默认口径：
  - `HLogInfo`
    - 低频关键状态变化
  - `HLogWarning`
    - 有 fallback、降级、跳过，但进程还能继续
  - `HLogError`
    - 当前操作失败，且需要调用方或用户关注
  - `HLogTrace`
    - 临时排障；默认不长期保留

### 注释

- 以下情况应该写注释：
  - 为什么这里不能按直觉写
  - 生命周期或调用顺序有硬约束
  - 所有权 / 借用关系不直观
  - command / undo / selection / persistence 之间有耦合约束
  - workaround 是为了解决某个已知问题
  - fallback / 兼容分支存在明确前提
  - 代码故意保持和看起来“更自然”的写法不一致
  - 这里如果改错，容易破坏跨模块行为
- 只写代码本身不容易表达的信息：
  - 为什么这样做
  - 生命周期约束
  - 所有权关系
  - workaround 背景
  - 失败语义
- 函数在以下情况建议写函数头注释：
  - 有副作用但名字看不出全部影响
  - 会修改多个 service / 全局状态
  - 调用顺序必须满足前置条件
  - 失败后对象是否仍有效不直观
- 成员变量在以下情况建议写行尾或上一行注释：
  - 生命周期依赖外部 owner
  - 只对某阶段有效
  - 缓存值和真源值容易混淆
- 注释优先解释约束和意图，不解释语法。
- 不要逐行翻译代码。
- 不要写“明显从代码就能看出来”的废话注释。
- TODO 要带目标，推荐：`TODO(editor): ...`
- FIXME 要能说明当前错误或风险，推荐：`FIXME(editor): ...`
- 不保留大段注释掉的旧代码。

## 10. 返回值与错误处理

- 只表达成功 / 失败：`bool`
- “可能没有结果，但不是错误”：`std::optional<T>`
- 需要带失败原因：用明确的 result struct 或等价方案

规则：

- `bool` 返回值必须能清楚回答：
  - `true` 是什么
  - `false` 是什么
  - 失败后对象状态是否仍然有效
- 不要用空对象同时表达“没有结果”和“失败”。
- 失败要么被上层明确处理，要么在合适边界记录；不要静默吞掉。

---

## 11. 持久化与序列化

- 持久化模型和运行态模型分离。
- 不把运行时句柄、缓存、UI 临时态直接写入持久化结构。
- JSON / 配置 key 用稳定语义命名，不要用临时缩写。
- 读取旧数据时必须有 fallback。
- 新增字段必须有默认值。
- 枚举优先序列化为稳定字符串，不要依赖枚举整数顺序。
- 路径必须明确：
  - workspace 相对
  - 项目相对
  - 绝对路径
- 同一序列化规则只保留一个主实现来源，不要多处复制。

---

## 12. 线程与帧边界

- 未明确声明线程安全的 Editor 类型，默认都不是线程安全的。
- `UIContext`、Panel、绝大多数 Editor service 默认只允许主线程访问。
- 不要跨帧持有只在当前帧有效的 UI 数据。
- 不要长期持有临时 payload、临时 rect、临时 observer 指针。
- attach / detach / initialize / shutdown / scene reload / layout reset 这类时序逻辑不要在多个地方各写一份补丁。

---

## 13. 验证规则

- 只改文档：检查索引是否同步。
- 改 Editor 代码：至少确保对应目标可编译。
- 改启动、布局、命令、序列化、UndoRedo、Scene workflow：必须做运行路径验证。
- 如果本次无法验证，交付说明里必须写清：
  - 哪些没验证
  - 为什么没验证
  - 风险在哪
- 接口、目录边界、工作流、构建方式变化后，要同步更新文档。

---

## 14. 临时代码

- 临时 trace、调试计数器、workaround、实验路径必须：
  - 作用域小
  - 容易识别
  - 容易删除
- 不要长期保留：
  - 永远打印的排障日志
  - 无人维护的 TODO
  - 注释掉的大块旧实现
  - 没有退出条件的 workaround

---

## 15. 改动范围约束

- 非必要不重命名文件。
- 非必要不移动文件。
- 非必要不重命名 public 类型、public 函数、配置 key、action id、panel id。
- 非必要不修改函数签名。重构任务明确要求改签名时除外。
- 非必要不做大规模格式化。
- 非必要不顺手修 unrelated code。
- 只改完成当前任务所需的最小范围。
- 如果一个改动开始扩散到多个无关模块，先停下来收窄范围。

---

## 16. 兼容性约束

- 非明确要求，不破坏现有序列化格式。
- 非明确要求，不修改已有配置 key。
- 非明确要求，不修改已有快捷键、菜单文案、action id、panel id、viewport id。
- 非明确要求，不改变已有用户可见行为。
- 修改持久化结构时必须保留 fallback。
- 任何可能影响旧数据、旧布局、旧配置的改动，都必须显式说明兼容性风险。

---

## 17. API 设计约束

- public 接口优先稳定；只在确实需要跨文件复用时新增 public API。
- helper 默认放 `.cpp`，不要轻易放进头文件。
- 参数超过 4 个时，优先考虑 options / desc struct。
- 不要用一个 `bool` 开关切换两种完全不同的语义。
- 不要让一个接口同时承担“查询 + 修改 + 持久化 + 通知”多种职责。
- 新增接口前，先检查是否已有相近能力可复用。

---

## 18. Lambda 与回调约束

- lambda 默认显式捕获，不要用 `[&]` 或 `[=]` 大范围捕获。
- 跨帧存活的回调禁止捕获局部引用。
- 回调里不要偷偷做重逻辑、持久化写盘或跨模块编排。
- 回调体过长时，提取为命名函数或 helper。
- 不要把复杂业务规则埋进注册回调的 lambda 里。

---

## 19. 容器与算法约束

- 查找优先 `find`，不要先 `count` 再 `find`。
- 能一次遍历完成的逻辑，不要拆成多次遍历。
- 删除元素时要明确处理迭代器失效。
- 默认优先可读写法，不要为了“更短”引入难读算法拼接。
- 只有在收益明确时才引入排序、复制、临时容器。
- 返回容器前先确认是否真的需要返回拥有值。

---

## 20. 分配与热路径约束

- 每帧路径避免反复创建临时 `std::string`、`std::vector`、`std::function`。
- 热路径避免隐藏堆分配。
- 热路径避免反复路径拼接、格式化字符串和容器拷贝。
- 能缓存的 UI 计算结果优先缓存。
- 不要为了代码简短在热路径引入额外对象构造。
- 进入每帧函数前，先判断这段逻辑是否会在 60FPS/120FPS 下持续执行。

---

## 21. 指针与空值约束

- 能返回引用时，不返回可空指针。
- 返回裸指针时，默认表示“借用，不拥有”。
- 不要用同一个空值同时表达多种语义。
- 不要让 `nullptr` 成为正常业务流程的常态分支。
- 如果空值是合法状态，命名或注释里要写清楚。
- 如果对象理论上初始化后必须存在，就不要继续到处判空。

---

## 22. 语言特性约束

- 非必要不引入复杂模板技巧。
- 非必要不引入宏生成业务代码。
- 不要在非允许场景为了省代码使用 `auto`（允许场景见第 7 节）。
- 不要为了省代码使用过度链式调用、嵌套 lambda 或晦涩表达式。
- 如果某个写法更“高级”但更难读，优先选择更直接的写法。

---

## 23. UI 约束

- UI 文案保持同一语言风格，不混杂。
- 新增按钮、菜单、popup、drag-drop payload 时，id 必须稳定。
- 稳定 id 优先集中定义，不要散落硬编码。
- UI 绘制函数不直接做持久化写盘。
- 面板开关状态变化走统一通路，不各自偷偷改。
- UI 输入结束后，如果涉及可撤销修改，进入 command 主链路，不直接写业务对象。

---

## 24. 交付说明约束

- AI 交付时必须说明：
  - 改了什么
  - 没改什么
  - 哪些地方刻意保持兼容
  - 哪些没验证
  - 风险在哪
- 如果为了收窄 diff 放弃了顺手修复，也要明确说明。
- 如果发现更大的结构问题但这次没处理，也要单独点出，不要偷偷扩大改动。

---

## 25. AI 改代码时的硬性检查

提交前至少检查：

- 我是不是又往 `EditorApplication` 塞了业务细节？
- 我是不是又往 `EditorContext` 加了宽依赖？
- 这次可撤销修改是不是走了 command 主链路？
- 新增依赖是不是本该非空，却被我写成可空裸指针？
- 我是不是复制了已有 helper，而不是抽公共实现？
- 头文件是否自洽，是否减少了不必要 include？
- 命名是否遵守当前规则，而不是引入新的风格？
- 我是不是显式写出了类型，而不是在非允许场景偷用了 `auto`？
- 参数和遍历是否避免了无意义拷贝，能否改成 `const&`？
- 这次改动是否保持了最小 diff，而不是顺手扩大范围？
- 我有没有无意改变序列化、配置、快捷键、id 或用户可见行为？
- 我有没有把重逻辑偷偷塞进 lambda、回调或 UI 绘制路径？
- 多个 bool 是否应该升级成枚举或 options struct？
- 日志是不是只在正确边界记录一次？
- 注释是不是在解释“为什么”，而不是翻译代码？
- 持久化字段是否稳定、是否有默认值和 fallback？
- 我留下的调试代码是否会成为长期噪音？
